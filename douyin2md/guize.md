

#### 一、 核心动作定义（六大基础动作）

1. **【过滤 (Filter)】**
   - **场景**：`work_labels` 中的标签为无意义数据、脏数据或明显噪声（如“测试”、“张三”、“null”、“乱码”、全标点）。
   - **逻辑**：系统依据黑名单/正则自动拦截，或人工审核删除，不进入标签库。
   - **作用**：保证标签库的清洁度，避免垃圾数据污染。
2. **【合并 (Merge)】**
   - **场景**：标签与系统中已存在的某个二级标签**字面完全一致**。
   - **逻辑**：系统自动匹配去重，不新建标签，直接归入该二级标签，计数累加。
   - **作用**：高效去重，这是最高频、最基础的自动化操作。
3. **【融入 (Assimilate)】**
   - **场景**：标签与系统中已存在的某个二级标签**语义相近**（如“单车”与“自行车”），但字面不同。
   - **逻辑**：不新建标签，将其作为同义词映射/归入到那个已存在的二级标签中。
   - **作用**：语义标准化，解决同义词与多重表达问题。
4. **【归类 (Categorize)】**
   - **场景**：标签明显属于某个已存在的一级分类，但该分类下没有合适的二级标签来收留它。
   - **逻辑**：归入该一级标签，并将该标签本身转化为新的二级标签。
   - **注**：若遇到同名冲突（即要生成的二级标签名 = 所在的一级标签名），需自动添加后缀或人工修改（如改为“一级标签名_相关”）。
   - **作用**：层级扩充，将有效的新标签精准挂在正确的分类树下。
5. **【创立 (Create)】**
   - **场景**：标签非常独特，无法归入任何现有分类，且数量规模较大（频次高），具备独立成类的价值。
   - **逻辑**：通过人工确认，新建一级标签（人工起名），并在其下创立对应的二级标签。
   - **作用**：体系扩张，用于动态接纳全新的业务领域。
6. **【暂存 (Suspend)】**
   - **场景**：标签独特、无法归类，且数量极少，或目前无法判断其长期价值。
   - **逻辑**：不新建正式结构，将其直接放入专用的“待分类”一级标签中，留待定期人工复盘。
   - **作用**：缓冲地带，避免因偶发长尾标签导致标签库碎片化。

#### 二、 系统映射与约束规则

1. **【容器规则】**：除“类别(Category)”和“作者(Author)”作为特殊的结构化容器外，所有常规的一级标签仅作为目录容器，不承载具体业务属性。所有的实际标签均作为二级标签存在。
2. **【命名排他规则】**：
   - 二级标签不允许和其所属的一级标签同名（如：一级标签是“AI”，其下的同义词标签应命名为“AI_技术”或“人工智能”，不能叫“AI”）。
   - 二级标签之间（即使在不同的一级目录下）也**不允许全局同名**，防止检索歧义。
3. **【动作闭环规则】**：标签生命周期中的动作仅限【过滤】【合并】【融入】【归类】【创立】【暂存】6种（修改、删除为后期维护动作，不在流水线内）。
4. **【强制规则干预 (Override)】**：
   - 允许通过外部配置文件（如 `rules.yaml`）随时添加强制干预规则，书写简单直白。
   - **示例**：`屏蔽融入：agent -> AI`；`强制融入：iphone -> 苹果`；`强制归类：苹果 -> 手机`。
   - **注入方式**：系统运行时通过读取外部文件生效；也可在终端运行结束时，通过命令行交互便捷录入新规则。
5. **【多重降级匹配策略】**：如果标签被人工规则拦截（如“agent”禁止融入“AI”），且受限于“容器规则”无法融入同名的一级标签“人工智能”，系统将自动降级寻找其他归宿；若无其他同义词匹配，则视作独立标签，执行【归类】（成为“人工智能”目录下的同级独立二级标签）。
6. **【同源去重与优先级】**：`filename_tags` 和 `tags` 里的标签，若与 `category`、`author` 完全重复，则优先保留 `category/author` 的值，剔除常规 tags 中的重复项。
7. **【LLM防歧义上下文提示】**：调用大模型验证时，必须**同时输入全量的一级类目和其下属的二级类目结构树**。例如提供“一级：水果->二级：苹果”和“一级：科技->二级：Apple”，以便LLM精准判断“苹果皮”与“苹果壳”的区别。

#### 三、 自动化流水线流程 (Pipeline)

**【阶段 0：准备与清洗环节】**
- **0.1 文本归一化**：提取所有 md 文档中的4个容器生成 `work_labels` 列表。统一转换为小写，去除首尾空格。
- **0.2 同源去重**：执行映射规则6，优先保留 Category/Author 的值，剔除 Tags 中的重复值。
- **0.3 黑名单过滤（执行【过滤】）**：匹配预设脏字库（如“测试”、“null”）、纯标点符号。匹配到的标签直接丢弃，从 `work_labels` 中脱离。

**【阶段 1：确定性匹配环节 (Exact Match)】**
- **1.1 结构化字段识别**：验证标签是否来自 `category` 及 `author`。如果是，直接归类到系统的【类别】和【作者】元数据库中，从 `work_labels` 中脱离。
- **1.2 强制规则拦截**：读取本地 `rules.yaml`，优先执行人工“强制融入”和“强制归类”动作；若匹配，则脱离。
- **1.3 字面量比对（执行【合并】）**：验证剩余标签是否与已存在的二级标签完全一致。如果一致，直接归入并累加频次，从 `work_labels` 中脱离。

**【阶段 2：AI 语义匹配与频次决策环节 (LLM Phase)】**
- **2.0 AI 提示词注入**：整理剩余标签库，并将当前的“一/二级标签树”、以及“禁止项（如agent不等于AI）”打包作为 System Prompt 提供给 LLM。
- **2.1 语义融入（执行【融入】）**：要求 LLM 验证当前标签是否属于某个已存在的二级标签的同义词。如果 LLM 返回确定的二级标签名，则执行融入并脱离（注：LLM返回的结果必须经过代码二次校验，确保未产生幻觉）。
- **2.2 树枝归类（执行【归类】）**：如果无法融入二级，要求 LLM 判断其是否属于某个已存在的一级分类。如果属于，则自动将该标签作为新的二级标签挂在该一级类目下，并脱离。*（注：若遇一级与二级同名冲突，程序自动加后缀以防终端高频打扰）。*
- **2.3 频次判定与最终收容**：统计当前 `work_labels` 中仍未脱离的“顽固标签”的出现频次，进行分支决策：
  - **2.3.1 高频创建（执行【创立】）**：频次 **>= 8**，触发终端拦截。一次性展示所有满足条件的标签。使用者可输入序号并为其起名创建新的【一级标签】。
    - *特例控制*：允许使用者输入特定指令（如 `skip` 或 回车跳过）拒绝创立。
  - **2.3.2 降级收留（执行【暂存】）**：频次 **< 8** 的标签，**以及在 2.3.1 中被人工拒绝创立的高频标签**，全部统一扔进预设的“待分类”一级标签容器中，完成本次流水线全流程。


 #### 四、  为了让系统更健壮，我为你补充以下几条策略和规则细节：

1. **【强制规则的管理方式】（回应你的第4点要求）：**
   - **建议方案**：使用一个独立的 `rules.yaml` 或 `rules.json` 文件进行维护，而不是写死在 python 源码里。
   - **书写格式示例**：
     ```yaml
     # 拦截规则 (不等于)
     not_assimilate:
       agent: ["AI", "人工智能"]  # agent不允许融入这两个标签
     # 强制融入规则
     force_assimilate:
       iphone: "苹果"          # 遇到iphone直接融入苹果，不经过LLM
     # 强制归类规则
     force_categorize:
       苹果: "手机"            # 苹果直接归入一级类目"手机"
     ```
   - **交互补充**：在终端运行每次结束时，可以提供一个指令 `add rule`，通过简单的命令行问答将新规则快速写入上述文件。

2. **【LLM幻觉防范规则】（补充规则）：**
   - LLM在执行【融入】(2.1)和【归类】(2.2)时，必须强制要求其输出的内容**只能从提供的已存在列表中选取**，不允许LLM自己创造不存在的二级或一级标签名称。如果不匹配，LLM必须输出 `None`。
